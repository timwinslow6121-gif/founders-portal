## Decisions Made
- Used `batch_alter_table` for all `ALTER TABLE` operations to ensure SQLite compatibility during local development while remaining PostgreSQL-compatible for production.
- Deliberately omitted FK constraint enforcement from migration 002 column definitions — SQLite ignores FK enforcement regardless, and adding named constraints caused batch mode errors. PostgreSQL enforces referential integrity via application-layer query scoping.
- agency_id columns NOT added as `db.Column` attributes in model classes — managed entirely by Alembic migrations to prevent Alembic from trying to add them again in migration 003 (which would cause a conflict).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] SQLite batch mode FK constraint naming requirement**
- **Found during:** Task 1 (migration 002 application)
- **Issue:** Using `sa.ForeignKey('agencies.id')` inline with `batch_alter_table.add_column` raised `ValueError: Constraint must have a name` in SQLite batch mode
- **Fix:** Removed inline FK from column definition. Used plain `sa.Integer()` column. FK relationship enforced at application layer via query scoping.
- **Files modified:** migrations/versions/f003d6f5cb50_add_agency_model_and_nullable_agency_id_.py
